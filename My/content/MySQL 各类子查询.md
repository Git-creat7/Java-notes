**💡一句话核心**

> 子查询就是“套娃”。先算括号里的，再算括号外的；如果括号里引用了括号外的变量，就是相关子查询。


子查询（Subquery）简单来说就是“嵌套查询”，即在一个 `SELECT`、`INSERT`、`UPDATE` 或 `DELETE` 语句内部又包含了一个 `SELECT` 语句。

你可以把子查询想象成编程里的**函数调用**：先运行内部的查询，拿到结果后，再作为参数交给外部的查询去处理。

按照返回结果的不同和出现位置的不同，子查询主要分为以下几类：

---

# 1. 按返回结果的“形状”分类

这是最常用的分类方式，决定了你在外部查询中能用什么符号对接。

|**类别**|**返回结果**|**常用操作符**|**场景举例**|
|---|---|---|---|
|**标量子查询**|单行单列（一个值）|`=`, `>`, `<`, `<>`|找出比平均工资高的人。|
|**列子查询**|多行单列（一列值）|`IN`, `ANY`, `ALL`|找出在“开发部”或“测试部”的员工 ID。|
|**行子查询**|单行多列（一条记录）|`(a, b) = (SELECT...)`|找出生日和性别都和张三一样的学生。|
|**表子查询**|多行多列（一张虚表）|`FROM` 之后做联查|将复杂的统计结果作为临时表再次过滤。|

---

# 2. 按执行逻辑分类（核心难点）


## ① 不相关子查询 (Independent Subquery)

- **逻辑：** 子查询独立运行，只运行一次，结果传给外部。
    
- **例子：** `SELECT * FROM emp WHERE salary > (SELECT AVG(salary) FROM emp);`
    
- **理解：** 先算出平均工资（比如 5000），然后外部查询查出所有大于 5000 的人。
    

## ② 相关子查询 (Correlated Subquery)

- **逻辑：** 子查询的执行**依赖于外部查询的每一行**。外部查询每查一行，子查询就要跑一次。
    
- **例子：** 找出每个部门里工资最高的人。
    
```SQL
    SELECT * FROM emp e1 
    WHERE salary = (SELECT MAX(salary) FROM emp e2 WHERE e2.dept_id = e1.dept_id);
```
    
- **理解：** 外部查询查到“张三”时，子查询会去跑一次“张三所在部门”的最高薪资。这种查询性能通常比不相关查询低。
    

---

# 3. 两个特殊的关键词：`EXISTS` vs `IN`

这是面试的高频考点，也是笔记里必记的内容。

- **`IN`：** 先执行子查询，把结果存在内存里，然后外部查询去比对。适合子查询结果集较小的情况。
    
- **`EXISTS`：** 外部查询每走一行，去子查询里看一眼“有没有数据”。它不关心返回了什么，只关心“有没有”。适合外部查询表小、子查询表大的情况。

# 自问自答
**Q: 子查询和 JOIN 哪个更好？**

**A:**
- **大多数情况下 JOIN 更好。** MySQL 优化器对 JOIN 的优化更成熟，能有效利用索引。

- **子查询更容易理解。** 逻辑直观，但在处理大数据量时容易产生临时表，导致性能下降。
    
- **结论：** 简单的查询用子查询没问题；复杂的、涉及大数据量的，优先重构成 JOIN。

```SQL
	-- 标量：查出比平均年龄大的学生
	SELECT * FROM students WHERE age > (SELECT AVG(age) FROM students);
	
	-- 列子查询：查出属于“计算机系”的学生
	SELECT name FROM students WHERE dept_id IN (SELECT id FROM departments WHERE name = 'CS');
```
