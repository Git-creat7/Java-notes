- 数据库管理系统 DBMS：位于用户/应用程序与操作系统之间，用于管理数据库的软件。

- 授权：`GRANT`；回收：`REVOKE`；SQL Server 还可用 `DENY` 明确拒绝。

## 1. 第一章：数据库基础

### 1.1 数据库、DBMS、DBS

**数据库 DB**：长期存储在计算机内、有组织、可共享的数据集合。数据库中存储的不只是数据本身，也存储数据之间的联系。

**数据库管理系统 DBMS**：位于用户/应用程序和操作系统之间的一层数据管理软件，负责数据定义、操纵、控制、维护和安全保护。

**数据库系统 DBS**：由数据库、数据库管理系统、应用系统、数据库管理员 DBA 和用户组成的整体。

**常考填空**：

- 数据库系统一般由数据库、应用系统、数据库管理系统、数据库管理员和用户构成。
- 数据库系统的核心是 DBMS。
- DBMS 能实现对数据的查询、插入、修改、删除，这叫数据操纵功能。

### 1.2 数据管理技术的发展

| 阶段 | 特点 | 数据独立性 |
|---|---|---|
| 人工管理 | 数据不保存，应用程序管理数据 | 很差 |
| 文件系统 | 数据以文件保存，程序与文件联系紧密 | 较差 |
| 数据库系统 | 数据统一管理、共享、独立 | 最高 |

数据库系统特点：数据结构化、共享性高、冗余度小、数据独立性高、统一管理和控制。

### 1.3 数据独立性

**物理独立性**：内模式改变时，模式尽量不变，应用程序也尽量不变。由“模式/内模式映像”保证。

**逻辑独立性**：模式改变时，外模式和应用程序尽量不变。由“外模式/模式映像”保证。

答题模板：

> 数据独立性是指应用程序与数据库中数据结构之间相互独立的程度。物理独立性指数据库物理存储结构改变时，逻辑结构和应用程序不必改变；逻辑独立性指数据库全局逻辑结构改变时，用户的外模式和应用程序不必改变。三级模式和两级映像保证了数据独立性。

## 2. 第二章：数据库系统结构与数据模型

### 2.1 数据模型三要素

1. 数据结构：描述数据对象及其联系。
2. 数据操作：查询、插入、删除、修改等。
3. 完整性约束：数据必须满足的约束条件。

### 2.2 概念模型和组织模型

- 概念模型：面向现实世界，与具体 DBMS 无关，常用 E-R 图表示。
- 组织模型/数据模型：层次模型、网状模型、关系模型等。
- 关系模型是用二维表表示实体和实体间联系的数据模型。

### 2.3 E-R 图符号

| 内容 | 图形 |
|---|---|
| 实体 | 矩形 |
| 属性 | 椭圆 |
| 联系 | 菱形 |
| 1:1、1:N、M:N | 在联系线上标注 |

实体之间联系：一对一、 一对多、 多对多。

### 2.4 关系模型三类完整性

1. **实体完整性**：主码属性不能为空，主码能唯一标识元组。对应 SQL：`PRIMARY KEY`。
2. **参照完整性**：外码要么为空，要么等于被参照表中某个主码值。对应 SQL：`FOREIGN KEY`。
3. **用户定义完整性**：根据应用语义自定义的约束。对应 SQL：`NOT NULL`、`UNIQUE`、`CHECK`、`DEFAULT` 等。

### 2.5 三级模式结构

| 层次 | 含义 |
|---|---|
| 外模式 | 用户看到的局部逻辑结构，又称用户视图 |
| 模式 | 数据库全体数据的全局逻辑结构 |
| 内模式 | 数据物理存储结构 |

两级映像：

- 外模式/模式映像：保证逻辑独立性。
- 模式/内模式映像：保证物理独立性。

简答模板：

> 数据库系统三级模式结构由外模式、模式和内模式组成。外模式是用户看到的局部逻辑结构；模式是数据库全体数据的全局逻辑结构；内模式是数据的物理存储结构。外模式/模式映像保证逻辑独立性，模式/内模式映像保证物理独立性。

## 3. 第三、四章：SQL 查询与更新速记

SQL Server 考试常用语法：

```sql
SELECT 列名
FROM 表名
WHERE 条件
GROUP BY 分组列
HAVING 分组条件
ORDER BY 排序列;
```

执行思路：`FROM/JOIN` -> `WHERE` -> `GROUP BY` -> `HAVING` -> `SELECT` -> `ORDER BY`。

常考：

- 单表查询：`WHERE`、`LIKE`、`BETWEEN`、`IN`、`IS NULL`
- 连接查询：`JOIN ... ON ...`
- 分组统计：`COUNT`、`AVG`、`MAX`、`MIN`、`SUM`
- 子查询：`IN`、`EXISTS`、`ALL`、`ANY`
- 更新：`INSERT`、`UPDATE`、`DELETE`
- 视图：`CREATE VIEW`
- 授权：`GRANT`、`REVOKE`

SQL Server 中文字符串建议写 `N'中文'`。

## 4. 第五章：视图

### 4.1 视图定义

视图是从一个或多个基本表导出的虚表。数据库中只保存视图定义，一般不单独存储视图对应的数据。

```sql
CREATE VIEW V_StudentScore AS
SELECT S.Sno, S.Sname, C.Cname, SC.Grade
FROM Student AS S
JOIN SC ON S.Sno = SC.Sno
JOIN Course AS C ON SC.Cno = C.Cno;
```

### 4.2 视图与基本表区别

| 比较 | 基本表 | 视图 |
|---|---|---|
| 数据来源 | 实际存储数据 | 从基本表导出 |
| 存储 | 存储真实数据 | 通常只存定义 |
| 修改 | 可直接增删改 | 受限制 |
| 用途 | 存储数据 | 简化查询、安全保护、逻辑独立性 |

### 4.3 视图优点

1. 简化用户操作。
2. 使用户从多角度看同一数据。
3. 提高逻辑数据独立性。
4. 提供安全保护，隐藏敏感字段。

## 5. 第六章：关系规范化理论

### 5.1 基本概念

- 函数依赖：若属性 X 的值能唯一确定属性 Y 的值，记作 `X -> Y`。
- 候选码：能唯一标识元组的最小属性集。
- 主属性：包含在任一候选码中的属性。
- 非主属性：不包含在任何候选码中的属性。
- 部分函数依赖：非主属性只依赖于复合码的一部分。
- 传递函数依赖：`X -> Y`，`Y -> Z`，则 `X -> Z`，且 Z 间接依赖 X。

### 5.2 范式速记

| 范式 | 条件 | 解决问题 |
|---|---|---|
| 1NF | 每个属性不可再分 | 表格原子化 |
| 2NF | 在 1NF 基础上，非主属性完全依赖于候选码 | 消除部分依赖 |
| 3NF | 在 2NF 基础上，非主属性不传递依赖于候选码 | 消除传递依赖 |

### 5.3 判断步骤

1. 写出关系模式 R(属性...)。
2. 找函数依赖。
3. 找候选码。
4. 判断是否 1NF。
5. 若候选码是复合码，看非主属性是否依赖于码的一部分；有则不是 2NF。
6. 看是否存在非主属性 -> 非主属性；有则不是 3NF。
7. 给出分解。

### 5.4 常用答题句

- 若关系中每个属性不可再分，则至少属于 1NF。
- 若每个非主属性都完全函数依赖于候选码，则属于 2NF。
- 若每个非主属性既不部分依赖于候选码，也不传递依赖于候选码，则属于 3NF。
- 规范化目的：减少数据冗余，避免插入异常、删除异常和修改异常。

## 6. 第七章：数据库保护

### 6.1 事务

事务是用户定义的一个数据库操作序列，是不可分割的工作单位。事务中的操作要么全做，要么全不做。

ACID：

| 特性 | 含义 |
|---|---|
| 原子性 Atomicity | 要么全做，要么全不做 |
| 一致性 Consistency | 事务执行前后数据库都处于一致状态 |
| 隔离性 Isolation | 并发事务之间互不干扰 |
| 持久性 Durability | 提交后结果永久保存 |

SQL Server 事务：

```sql
BEGIN TRANSACTION;
-- SQL 操作
COMMIT TRANSACTION;   -- 提交
-- 或
ROLLBACK TRANSACTION; -- 回滚
```

### 6.2 并发控制

不加控制会出现：

1. 丢失修改：两个事务修改同一数据，后写覆盖前写。
2. 读脏数据：读到其他事务未提交的数据。
3. 不可重复读：同一事务两次读取同一数据结果不同。

并发控制主要方法：封锁机制。

| 锁 | 名称 | 作用 |
|---|---|---|
| S 锁 | 共享锁/读锁 | 可读，不可写；多个事务可同时加 S 锁 |
| X 锁 | 排它锁/写锁 | 可读可写；其他事务不能再加任何锁 |

若事务 T 对 A 加 S 锁，其他事务只能对 A 再加 S 锁，不能加 X 锁。  
若事务 T 对 A 加 X 锁，其他事务不能对 A 加任何锁。

### 6.3 三级封锁协议

| 协议 | 要求 | 可解决 |
|---|---|---|
| 一级封锁协议 | 修改数据前加 X 锁，事务结束释放 | 防止丢失修改 |
| 二级封锁协议 | 一级 + 读数据前加 S 锁，读完释放 | 防止丢失修改、读脏数据 |
| 三级封锁协议 | 一级 + 读数据前加 S 锁，事务结束释放 | 防止丢失修改、读脏数据、不可重复读 |

### 6.4 两段锁协议 2PL

事务分两个阶段：

1. 扩展阶段：只申请锁，不释放锁。
2. 收缩阶段：只释放锁，不再申请锁。

结论：遵守两段锁协议的调度一定是可串行化调度。两段锁协议是可串行化的充分条件，不是必要条件。

### 6.5 活锁和死锁

**活锁**：某事务一直等待，总被其他事务抢先获得资源。可用先来先服务策略避免。

**死锁**：两个或多个事务相互等待对方释放锁，导致都无法继续执行。

预防死锁：

1. 一次封锁法：事务一次申请全部要用的数据锁。
2. 顺序封锁法：预先规定数据对象封锁顺序，所有事务按同一顺序加锁。

检测死锁：超时法、事务等待图法。  
解除死锁：选择代价小的事务撤销，释放锁，并恢复其修改。

### 6.6 数据库恢复

故障类型：

- 事务故障
- 系统故障，也称软故障
- 介质故障，也称硬故障
- 计算机病毒等

恢复基本原理：利用冗余数据重建数据库。平时要做好：

1. 数据转储/备份。
2. 登记日志文件。

日志文件作用：记录事务对数据库的更新操作，用于事务故障恢复、系统故障恢复，并协助后备副本进行介质故障恢复。

**先写日志，后写数据库**：若先改数据库后写日志，系统故障可能导致数据库已变但日志无记录，无法恢复。

恢复策略：

| 故障 | 策略 |
|---|---|
| 事务故障 | UNDO 撤销该事务 |
| 系统故障 | UNDO 未提交事务，REDO 已提交事务 |
| 介质故障 | 重装备份副本，再用日志 REDO 已提交事务 |

## 7. 第八章：数据库设计

### 7.1 数据库设计步骤

1. 需求分析。
2. 概念结构设计。
3. 逻辑结构设计。
4. 物理结构设计。
5. 数据库实施。
6. 数据库运行和维护。

与具体 DBMS 无关：需求分析、概念结构设计。  
与具体 DBMS 密切相关：物理结构设计。

### 7.2 E-R 图转关系表

| 联系类型 | 转换方法 |
|---|---|
| 实体 | 每个实体转换为一个关系表，实体属性变字段，实体码变主键 |
| 1:1 | 可将任一方主键加入另一方作为外键，也可单独建联系表 |
| 1:N | 将 1 方主键加入 N 方作为外键；联系属性也放入 N 方 |
| M:N | 单独建立联系表，包含两边实体主键作为外键，通常联合组成主键；联系属性放入该联系表 |

## 8. 第十一章：数据库安全管理

### 8.1 核心概念

- 登录名 Login：用于连接 SQL Server 实例。
- 数据库用户 User：某数据库内部的访问主体，通常映射到登录名。
- 权限 Permission：允许用户执行的操作，如 `SELECT`、`INSERT`、`UPDATE`。
- 角色 Role：权限集合，可将用户加入角色以便统一授权。

关系：登录名进入服务器，数据库用户访问具体数据库，角色用于批量管理权限。

### 8.2 安全控制措施

1. 用户标识和鉴别。
2. 存取控制：授权和合法权限检查。
3. 视图机制：隐藏敏感数据。
4. 审计：记录用户操作。
5. 数据加密。
6. 操作系统安全保护。

### 8.3 授权与回收

```sql
GRANT SELECT, INSERT ON Student TO User1;
REVOKE SELECT ON Student FROM User1;
```

授予所有用户：

```sql
GRANT SELECT ON Student TO public;
```

SQL Server 明确拒绝某用户：

```sql
DENY SELECT ON Student TO Gary;
```
