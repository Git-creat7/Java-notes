# 综合题答题模板（数据库设计 + 范式 + 事务并发恢复 + 安全）

> 目标：专门应对综合题 40 分。  
> 最常见组合：给业务场景，要求画 E-R 图、转关系表、写主外键、判范式、设计约束/索引；或问事务、并发、恢复、安全管理。

---

## 0. 综合题通用作答顺序

遇到大题先按这个顺序写，不容易漏分：

1. **找实体**：名词一般是实体，如学生、课程、商店、商品、医生、病人。
2. **找属性**：题目列出的信息一般是属性。
3. **找联系**：动词或业务关系一般是联系，如选课、供应、销售、聘用、借阅。
4. **标联系类型**：1:1、1:N、M:N。
5. **画/描述 E-R 图**：实体、属性、联系、联系类型。
6. **转关系模式**：实体表 + 必要的联系表。
7. **写主键/外键**：每张表必须写。
8. **判范式**：一般至少到 2NF/3NF。
9. **写完整性约束**：实体完整性、参照完整性、用户定义完整性。
10. **若问物理设计**：写索引字段和理由。

## 1. E-R 图转关系表模板

### 1.1 实体转换

模板：

```text
实体名(主键, 属性1, 属性2, ...)
主键：xxx
```

例：

```text
学生(学号, 姓名, 性别, 年龄, 专业, 院系)
主键：学号
```

### 1.2 1:1 联系转换

方法一：任选一方加入另一方主键作为外键，并加入联系属性。

```text
A(A号, A属性, B号, 联系属性)
B(B号, B属性)
```

方法二：联系单独成表。

```text
联系表(A号, B号, 联系属性)
主键：A号 或 B号 或 (A号, B号)
外键：A号 引用 A，B号 引用 B
```

### 1.3 1:N 联系转换

把 1 方主键加入 N 方作为外键；联系属性也加入 N 方。

```text
部门(部门号, 部门名)
职工(职工号, 姓名, 部门号, 入职时间)
```

- 部门是 1 方。
- 职工是 N 方。
- `部门号` 放入职工表作为外键。

### 1.4 M:N 联系转换

必须单独建立联系表，包含两边主键作为外键，通常联合组成主键；联系属性放在联系表中。

```text
学生(学号, 姓名, ...)
课程(课程号, 课程名, ...)
选课(学号, 课程号, 选课时间, 成绩)
```

- 选课表主键：`(学号, 课程号, 选课时间)` 或 `(学号, 课程号)`，按题意决定。
- 外键：`学号` 引用学生，`课程号` 引用课程。

## 2. 数据库设计步骤模板

### 2.1 六个阶段

1. 需求分析。
2. 概念结构设计。
3. 逻辑结构设计。
4. 物理结构设计。
5. 数据库实施。
6. 数据库运行和维护。

### 2.2 常考问法

**问：哪个阶段与具体 DBMS 无关？**

答：需求分析和概念结构设计与具体 DBMS 无关。概念结构设计得到 E-R 图，面向现实世界，不依赖具体数据库管理系统。

**问：哪个阶段与具体 DBMS 密切相关？**

答：物理结构设计与具体 DBMS 密切相关，因为索引、存储结构、存取路径等依赖具体 DBMS。

**问：数据字典是什么？**

答：数据字典是系统中各类数据描述的集合，是需求分析阶段进行详细数据收集和数据分析的主要成果，通常包括数据项、数据结构、数据流、数据存储和处理过程。

## 3. 往年综合题模板一：成绩管理系统

### 3.1 题干抽象

需要存储：

1. 学生：学号、姓名、性别、年龄、专业、院系。
2. 课程：课程编号、课程名称、学时、任课老师。
3. 选课：学号、课程编号、选课时间、分数。
4. 一个学生可选多门课程，一门课程可被多个学生选修，所以学生与课程是 M:N。
5. 每个学生每门课程在一个学期/选课时间只能选一次。
6. 成绩百分制，学时不能超过 100。

### 3.2 E-R 图文字描述

```text
学生实体：学号、姓名、性别、年龄、专业、院系；主属性/码为学号。
课程实体：课程编号、课程名称、学时、任课教师；主属性/码为课程编号。
学生与课程之间存在“选课”联系，联系类型为 M:N。
选课联系属性：选课时间、分数。
```

若要画图：学生矩形 —— 菱形“选课” —— 课程矩形，两边标 M:N；选课连接属性“选课时间、分数”。

### 3.3 逻辑结构

```text
学生表(学号, 姓名, 性别, 年龄, 专业, 院系)
课程表(课程编号, 课程名称, 学时, 任课教师)
选课表(学号, 课程编号, 选课时间, 分数)
```

### 3.4 主键、候选键、外键

```text
学生表：
候选键：学号
主键：学号
外键：无

课程表：
候选键：课程编号
主键：课程编号
外键：无

选课表：
候选键：(学号, 课程编号, 选课时间)
主键：(学号, 课程编号, 选课时间)
外键：学号 引用 学生表(学号)，课程编号 引用 课程表(课程编号)
```

若题目没有“同一课程可多次选课/按学期选课”的条件，也可把 `(学号, 课程编号)` 作为选课表主键。这里题目有选课时间，所以更稳写 `(学号, 课程编号, 选课时间)`。

### 3.5 范式判断

推荐答法：

```text
学生表中每个属性不可再分，属于 1NF。若认为“学号 -> 专业，专业 -> 院系”，则存在传递依赖“学号 -> 专业 -> 院系”，所以学生表最高为 2NF，不属于 3NF。可分解为：学生(学号, 姓名, 性别, 年龄, 专业)，专业(专业, 院系)。

课程表主键为课程编号，非主属性课程名称、学时、任课教师都完全依赖于课程编号，且不存在非主属性对主键的传递依赖，属于 3NF。

选课表主键为(学号, 课程编号, 选课时间)，非主属性分数依赖于整个主键，不存在部分依赖和传递依赖，属于 3NF。
```

如果老师不强调“专业 -> 院系”，也可写学生表属于 3NF。但为了稳妥，可加一句：“若一个专业唯一属于一个院系，则存在传递依赖，此时应分解。”

### 3.6 完整性约束

```text
实体完整性：学生表学号不能为空且唯一；课程表课程编号不能为空且唯一；选课表联合主键不能为空且唯一。
参照完整性：选课表学号必须引用学生表已有学号；选课表课程编号必须引用课程表已有课程编号。
用户定义完整性：性别只能为男或女；年龄在 15 到 45 之间；学时在 1 到 100 之间；分数在 0 到 100 之间。
```

### 3.7 索引设计

题：选课时间和分数经常作为筛选条件进行查询。

答：

```text
可在选课表的选课时间字段和分数字段上建立索引。原因是这两个字段经常作为 WHERE 筛选条件，建立索引可加快按选课时间和分数检索的速度。但索引会占用额外空间，并降低插入、删除和修改的速度。
```

SQL Server：

```sql
CREATE INDEX IX_选课表_选课时间 ON 选课表(选课时间);
CREATE INDEX IX_选课表_分数 ON 选课表(分数);
```

## 4. 往年综合题模板二：商店-商品-职工

### 4.1 题型 A：商店与职工 1:N，商店与商品 M:N

题干常见信息：

- 商店：商店编号、商店名、地址。
- 商品：商品号、商品名、规格、单价。
- 职工：职工编号、姓名、性别、业绩。
- 每个商店有多个职工，每个职工只能在一个商店工作，商店聘用职工有聘期、月薪。
- 每个商店销售多种商品，每种商品可在多个商店销售，销售联系有月销售量。

转换：

```text
商店(商店编号, 商店名, 地址)
主键：商店编号

商品(商品号, 商品名, 规格, 单价)
主键：商品号

职工(职工编号, 姓名, 性别, 业绩, 商店编号, 聘期, 月薪)
主键：职工编号
外键：商店编号 引用 商店(商店编号)

销售(商店编号, 商品号, 月销售量)
主键：(商店编号, 商品号)
外键：商店编号 引用 商店(商店编号)，商品号 引用 商品(商品号)
```

### 4.2 题型 B：供应商-商店-商品

题干常见信息：

- 供应商可供应多种商品，每种商品可向多个供应商订购，供应联系有月供应量。
- 商店可销售多种商品，每种商品可在多个商店销售，销售联系有月计划数。

转换：

```text
供应商(供应商编号, 供应商名, 地址)
主键：供应商编号

商店(商店号, 商店名, 地址)
主键：商店号

商品(商品号, 商品名, 规格, 单价)
主键：商品号

供应(供应商编号, 商品号, 月供应量)
主键：(供应商编号, 商品号)
外键：供应商编号 引用 供应商，商品号 引用 商品

销售(商店号, 商品号, 月计划数)
主键：(商店号, 商品号)
外键：商店号 引用 商店，商品号 引用 商品
```

## 6. 往年综合题模板四：医院病房管理

题干常见信息：

- 科室：科名、科地址、科电话。
- 病房：病房号、床位号、所属科室。
- 医生：工作证号、姓名、职称、所属科室、年龄。
- 病人：病历号、姓名、性别、诊断、主管医生、病房号。

关系模式：

```text
科室(科名, 科地址, 科电话)
主键：科名

病房(科名, 病房号, 床位号)
主键：(科名, 病房号)
外键：科名 引用 科室(科名)

医生(工作证号, 姓名, 职称, 科名, 年龄)
主键：工作证号
外键：科名 引用 科室(科名)

病人(病历号, 姓名, 性别, 诊断, 主管医生工作证号, 科名, 病房号)
主键：病历号
外键：主管医生工作证号 引用 医生(工作证号)
外键：(科名, 病房号) 引用 病房(科名, 病房号)
```

说明：题库中有时写 `病房(病房号, 床位号, 科室号)`，但若病房号只在科室内唯一，主键应写 `(科名/科室号, 病房号)` 更严谨。

## 8. 范式综合题模板

### 8.1 答题步骤

```text
第一步：写出关系模式 R(...)
第二步：根据题意写出函数依赖集 F
第三步：求候选码
第四步：判断 1NF、2NF、3NF
第五步：若不满足，给出分解方案
```

### 8.2 2NF 判断模板

```text
R 已满足 1NF，因为各属性不可再分。
R 的候选码为 (...)。
由于存在非主属性 A 只依赖于候选码的一部分 B，即 B -> A，存在部分函数依赖，所以 R 不满足 2NF。
可分解为 R1(B, A) 和 R2(原候选码及其余属性)。
```

例：

```text
选课临时表(学号, 课程号, 姓名, 课程名, 成绩)
函数依赖：学号 -> 姓名；课程号 -> 课程名；(学号, 课程号) -> 成绩
候选码：(学号, 课程号)
姓名部分依赖于学号，课程名部分依赖于课程号，所以不满足 2NF。
分解：学生(学号, 姓名)，课程(课程号, 课程名)，选课(学号, 课程号, 成绩)。
```

### 8.3 3NF 判断模板

```text
R 已满足 2NF。
但存在候选码 -> 非主属性 -> 非主属性 的传递依赖，即 X -> Y，Y -> Z，所以 R 不满足 3NF。
可分解为 R1(X, Y) 和 R2(Y, Z)。
```

例：

```text
学生(学号, 姓名, 专业, 院系)
函数依赖：学号 -> 姓名, 专业；专业 -> 院系
存在 学号 -> 专业 -> 院系，所以院系传递依赖于学号，不满足 3NF。
分解：学生(学号, 姓名, 专业)，专业(专业, 院系)。
```

### 8.4 范式结论句

- 若每个属性不可再分，则 R 属于 1NF。
- 若 R 属于 1NF，且所有非主属性完全依赖于候选码，则 R 属于 2NF。
- 若 R 属于 2NF，且不存在非主属性对候选码的传递依赖，则 R 属于 3NF。

## 9. 事务与并发控制综合题模板

### 9.1 事务 ACID 简答模板

```text
事务是用户定义的一个数据库操作序列，是不可分割的工作单位。事务具有 ACID 特性：
1. 原子性：事务中的操作要么全做，要么全不做。
2. 一致性：事务执行前后数据库都应保持一致状态。
3. 隔离性：多个事务并发执行时互不干扰。
4. 持久性：事务一旦提交，其结果永久保存。
```

### 9.2 并发异常模板

```text
如果并发操作不加控制，可能造成数据不一致，主要包括：
1. 丢失修改：两个事务同时修改同一数据，后一个修改覆盖前一个修改。
2. 读脏数据：一个事务读取了另一个事务尚未提交的数据。
3. 不可重复读：同一事务两次读取同一数据，由于其他事务修改并提交，导致两次读取结果不同。
```

### 9.3 封锁协议模板

```text
并发控制常用封锁机制。S 锁为共享锁，多个事务可同时对同一数据加 S 锁，但不能再加 X 锁；X 锁为排它锁，一个事务加 X 锁后，其他事务不能再加任何锁。

一级封锁协议：修改数据前加 X 锁，事务结束释放，可防止丢失修改。
二级封锁协议：一级基础上，读数据前加 S 锁，读完释放，可防止丢失修改和读脏数据。
三级封锁协议：一级基础上，读数据前加 S 锁，事务结束释放，可防止丢失修改、读脏数据和不可重复读。
```

### 9.4 两段锁协议模板

```text
两段锁协议要求事务分两个阶段加锁和解锁：第一阶段为扩展阶段，只申请和获得锁，不释放锁；第二阶段为收缩阶段，只释放锁，不再申请新锁。遵守两段锁协议的并发调度一定是可串行化的，因此两段锁协议是可串行化调度的充分条件。
```

### 9.5 死锁模板

```text
死锁是指两个或多个事务在执行过程中，由于竞争资源而相互等待对方释放锁，若无外力作用都无法继续执行。
预防死锁的方法主要有一次封锁法和顺序封锁法。一次封锁法要求事务一次申请全部要使用的数据锁；顺序封锁法要求所有事务按预先规定的顺序对数据对象加锁。
死锁检测方法有超时法和事务等待图法。检测到死锁后，通常选择代价较小的事务撤销，释放其持有的锁，并利用恢复机制撤销该事务已做的修改。
```

## 10. 数据库备份与恢复综合题模板

### 10.1 恢复基本原理

```text
数据库恢复的基本原理是利用冗余。系统平时通过数据转储和登记日志文件保存冗余信息，当数据库发生故障时，利用备份副本和日志文件将数据库恢复到一致状态。
```

### 10.2 为什么要先写日志后写数据库

```text
登记日志文件时必须先写日志文件，后写数据库。因为如果先修改数据库而日志尚未记录，系统发生故障后就无法知道该修改的具体内容，也就无法进行撤销或重做，从而不能保证数据库恢复到一致状态。
```

### 10.3 系统故障恢复模板

```text
系统故障发生后，数据库缓冲区内容丢失，但磁盘数据库未被破坏。恢复时应根据日志文件找出故障发生时未提交的事务和已提交但可能未写入数据库的事务。对未提交事务执行 UNDO，撤销其影响；对已提交事务执行 REDO，重做其更新，使数据库恢复到一致状态。
```

### 10.4 介质故障恢复模板

```text
介质故障会破坏磁盘上的物理数据库。恢复步骤为：
1. 重装备份/转储副本，使数据库恢复到备份时的一致状态。
2. 利用日志文件找出备份后已经提交的事务。
3. 对这些已提交事务执行 REDO，重做其更新，使数据库恢复到故障前某一一致状态。
```

### 10.5 日志文件内容与作用

```text
日志文件记录事务开始、提交、回滚以及每次更新操作的相关信息，包括事务标识、操作类型、操作对象、更新前值和更新后值等。日志文件用于事务故障恢复、系统故障恢复，并协助后备副本进行介质故障恢复。
```

## 11. 安全管理综合题模板

### 11.1 登录名、用户、权限、角色关系

```text
登录名是 SQL Server 服务器级主体，用于登录服务器实例；数据库用户是数据库级主体，用于访问具体数据库，通常由登录名映射而来；权限是用户或角色对数据库对象可执行的操作，如 SELECT、INSERT、UPDATE、DELETE；角色是权限的集合，可以把多个用户加入角色统一授权。
```

### 11.2 数据库安全性设计模板

```text
作为数据库设计人员，可以从以下方面考虑数据库安全性：
1. 用户标识和鉴别：通过用户名、口令等方式确认用户身份。
2. 存取控制：通过 GRANT、REVOKE 等语句授予或回收权限，确保用户只能访问被授权的数据。
3. 角色管理：把权限授予角色，再把用户加入角色，简化权限管理。
4. 视图机制：为不同用户定义不同视图，隐藏敏感字段或记录。
5. 数据加密：对存储和传输中的敏感数据加密。
6. 审计：记录用户对数据库的重要操作，便于追踪非法访问。
7. 操作系统安全控制：防止数据库文件被非法复制、删除或破坏。
```

### 11.3 授权综合题模板

题：将某表查询权、某字段修改权授予用户。

```sql
GRANT SELECT ON 表名 TO 用户名;
GRANT UPDATE(字段名) ON 表名 TO 用户名;
```

题：回收权限。

```sql
REVOKE SELECT ON 表名 FROM 用户名;
REVOKE UPDATE(字段名) ON 表名 FROM 用户名;
```

题：只允许看部分数据。

```sql
CREATE VIEW 视图名 AS
SELECT 需要给用户看的列
FROM 基本表
WHERE 条件;

GRANT SELECT ON 视图名 TO 用户名;
```

题：授予所有人但排除 Gary。

SQL Server 可写：

```sql
GRANT SELECT ON 对象名 TO public;
DENY SELECT ON 对象名 TO Gary;
```

或者说明：建立不包含 Gary 的角色，对该角色授权。

## 12. 综合题卷面加分写法

### 12.1 写主外键时不要只写“有外键”

推荐格式：

```text
选课表：
主键：(学号, 课程编号, 选课时间)
外键：学号 引用 学生表(学号)；课程编号 引用 课程表(课程编号)
```

### 12.2 写完整性约束时分三类

```text
实体完整性：主键非空且唯一。
参照完整性：外键必须引用已存在的主键值。
用户定义完整性：性别取男/女，年龄范围，成绩 0~100，学时不超过 100。
```

### 12.3 写范式时必须有“原因”

不要只写“属于 3NF”。应写：

```text
该关系主键为课程编号，所有非主属性都完全依赖于课程编号，且不存在非主属性对主键的传递依赖，所以属于 3NF。
```

### 12.4 写索引时必须有“字段 + 理由”

```text
在选课时间、分数上建索引，因为它们经常作为查询筛选条件，可提高检索速度。但索引会占用存储空间，并可能降低更新速度。
```

### 12.5 画不出图时可用文字 E-R 描述保分

如果来不及画图，至少写：

```text
实体：学生、课程。
属性：学生(学号, 姓名, ...)，课程(课程编号, 课程名称, ...)。
联系：学生与课程之间存在选课联系，类型为 M:N，联系属性为选课时间、分数。
```
