## Java 参数校验（Bean Validation）

### 常用注解

| 注解                 | 说明                                  |
| ------------------ | ----------------------------------- |
| `@NotNull`         | 值不能为 `null`，但允许空字符串                 |
| `@NotEmpty`        | 值不能为 `null` 且长度 > 0（适用于字符串、集合）      |
| `@NotBlank`        | 值不能为 `null`，且去除空白字符后长度 > 0（仅适用于字符串） |
| `@Size(min, max)`  | 字符串长度 / 集合大小必须在 `[min, max]` 范围内    |
| `@Min(value)`      | 数值最小值                               |
| `@Max(value)`      | 数值最大值                               |
| `@Pattern(regexp)` | 字符串必须匹配正则表达式                        |
| `@Email`           | 字符串必须是合法 Email 格式                   |

### 示例

```java
@NotBlank(message = "登录账号不能为空")
@Size(min = 2, max = 16, message = "账号长度必须在2-16之间")
private String username;

@NotBlank(message = "密码不能为空")
private String password;
```

> 注意：同一字段重复标注相同注解（如两个 `@NotBlank`）是多余的，只保留一个即可。

### 三者区别

```
@NotNull  → null 不通过，"" 通过，" " 通过
@NotEmpty → null 不通过，"" 不通过，" " 通过
@NotBlank → null 不通过，"" 不通过，" " 不通过  ✅ 最严格，推荐用于字符串字段

name 是 String，用户传 "" 空字符串你也不该接受。
String 字段一律用 @NotBlank，@NotNull 留给 Long/Integer/对象这种没法 blank 的类型。
```

### 使用步骤

1. **引入依赖**（Spring Boot 已内置）

2. **在 DTO/实体类字段上加注解**（如上示例）

3. **在 Controller 方法参数上加 `@Valid` 或 `@Validated`**

   ```java
   @PostMapping("/login")
   public Result login(@RequestBody @Valid LoginDTO dto) {
       // ...
   }
   ```

4. **全局异常处理**（捕获校验失败异常）

   ```java
   @RestControllerAdvice
   public class GlobalExceptionHandler {

       @ExceptionHandler(MethodArgumentNotValidException.class)
       public Result handleValidException(MethodArgumentNotValidException e) {
           String msg = e.getBindingResult().getFieldErrors().stream()
               .map(FieldError::getDefaultMessage)
               .collect(Collectors.joining("; "));
           return Result.fail(msg);
       }
   }
   ```

### `@Valid` vs `@Validated`

| | `@Valid` | `@Validated` |
|---|---|---|
| 来源 | JSR-303 标准 | Spring 扩展 |
| 分组校验 | 不支持 | 支持 |
| 嵌套校验 | 支持（加在字段上） | 支持 |
| 常用场景 | 一般校验 | 需要分组时使用 |

# Spring Security
**Spring Security 本质就是“拦截请求 + 校验身份 + 判断权限”**

Spring Security 是 Spring 生态中最核心的安全框架，用于为 Java / Spring Boot 应用提供：

- 用户认证（Authentication）
- 权限控制（Authorization）
- 防御常见攻击（CSRF、Session Fixation 等）
- JWT / OAuth2 / SSO / LDAP 集成

通常用于：

- 后端 REST API
- 管理系统
- 企业级系统
- 微服务认证中心
- 单点登录（SSO）

---

## 核心概念

| 概念                   | 说明                                     |
| -------------------- | -------------------------------------- |
| **认证**               | 验证"你是谁"，即登录验证                          |
| **授权**               | 验证"你能做什么"，即权限控制                        |
| `SecurityContext`    | 保存当前用户认证信息的容器（线程级别）                    |
| `Authentication`     | 表示已认证的用户信息（主体 + 凭证 + 权限）               |
| `UserDetails`        | 框架用于描述用户的接口，需自行实现                      |
| `UserDetailsService` | 根据用户名加载用户信息的接口，核心扩展点                   |
| `PasswordEncoder`    | 密码加密/校验接口，推荐使用 `BCryptPasswordEncoder` |

## 认证流程（表单登录 / JWT）

```Java
请求 → UsernamePasswordAuthenticationFilter
         ↓
     AuthenticationManager（ProviderManager）
         ↓
     AuthenticationProvider（DaoAuthenticationProvider）
         ↓
     UserDetailsService.loadUserByUsername()
         ↓
     PasswordEncoder.matches()
         ↓
     认证成功 → 存入 SecurityContextHolder
```

## Authentication（认证）

“你是谁？”

验证用户身份。

例如：

- 用户名密码登录
- JWT Token
- OAuth2 登录
- 微信/Google/GitHub 登录

例如：

```Java
	@Bean
	public UserDetailsService userDetailsService() {
	    UserDetails user = User.withDefaultPasswordEncoder()
	            .username("admin")
	            .password("123456")
	            .roles("ADMIN")
	            .build();
	
	    return new InMemoryUserDetailsManager(user);
	}
```

## Authorization（授权）

“你能访问什么？”

比如：

- 普通用户只能访问 `/user`
- 管理员才能访问 `/admin`

示例：

```Java
	http.authorizeHttpRequests(auth -> auth
	        .requestMatchers("/admin/**").hasRole("ADMIN")
	        .requestMatchers("/user/**").hasRole("USER")
	        .anyRequest().authenticated()
	);
```

## 工作原理
核心思想：

> 基于 Filter（过滤器链）

```
	请求
	 ↓
	Security Filter Chain
	 ↓
	认证
	 ↓
	权限检查
	 ↓
	Controller
```

**核心部件**

| 组件                      | 作用     |
| ----------------------- | ------ |
| `SecurityFilterChain`   | 安全过滤器链 |
| `Authentication`        | 当前登录用户 |
| `UserDetailsService`    | 查询用户   |
| `PasswordEncoder`       | 密码加密   |
| `SecurityContextHolder` | 保存登录信息 |
```Java
	@Bean
	public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
	
	    http
	        .csrf(csrf -> csrf.disable())
	        .authorizeHttpRequests(auth -> auth
	            .anyRequest().authenticated()
	        )
	        .formLogin(Customizer.withDefaults());
	
	    return http.build();
	}
```

## UserDetails & UserDetailsService

Spring Security 通过 `UserDetailsService` 加载用户，需要自行实现。

### 实现 UserDetails（封装用户信息）

```java
@Data
@AllArgsConstructor
@NoArgsConstructor
public class LoginUser implements UserDetails {

    private User user;

    @Override
    public Collection<? extends GrantedAuthority> getAuthorities() {
        // 将权限字符串转为 GrantedAuthority 集合
        return user.getPermissions().stream()
                .map(SimpleGrantedAuthority::new)
                .collect(Collectors.toList());
    }

    @Override public String getPassword()               { return user.getPassword(); }
    @Override public String getUsername()               { return user.getUsername(); }
    @Override public boolean isAccountNonExpired()      { return true; }
    @Override public boolean isAccountNonLocked()       { return true; }
    @Override public boolean isCredentialsNonExpired()  { return true; }
    @Override public boolean isEnabled()                { return true; }
}
```

### 实现 UserDetailsService（查询数据库）

```java
@Service
public class UserDetailsServiceImpl implements UserDetailsService {

    @Autowired
    private UserMapper userMapper;

    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        User user = userMapper.selectByUsername(username);
        if (user == null) {
            throw new UsernameNotFoundException("用户名或密码错误");
        }
        return new LoginUser(user);
    }
}
```

## JWT 集成（前后端分离）

前后端分离项目一般关闭 Session，使用 JWT 做无状态认证。

### 整体流程

```
登录请求
  ↓
手动调用 AuthenticationManager.authenticate()
  ↓
认证通过 → 生成 JWT → 返回给前端
  ↓
后续请求携带 JWT（Header: Authorization: Bearer xxx）
  ↓
JwtAuthenticationFilter 解析 Token → 写入 SecurityContextHolder
  ↓
Controller 正常处理
```

### 登录接口

```java
@RestController
public class LoginController {

    @Autowired
    private AuthenticationManager authenticationManager;

    @PostMapping("/login")
    public Result login(@RequestBody @Valid LoginDTO dto) {
        UsernamePasswordAuthenticationToken authToken =
                new UsernamePasswordAuthenticationToken(dto.getUsername(), dto.getPassword());

        // 触发认证，内部会调用 UserDetailsService + PasswordEncoder
        Authentication authentication = authenticationManager.authenticate(authToken);

        LoginUser loginUser = (LoginUser) authentication.getPrincipal();
        String jwt = JwtUtil.createJWT(loginUser.getUser().getId().toString());
        return Result.ok(jwt);
    }
}
```

### JWT 认证过滤器

```java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    @Autowired
    private RedisCache redisCache;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) throws ServletException, IOException {
        String token = request.getHeader("Authorization");
        if (!StringUtils.hasText(token)) {
            // 没有 token，放行（后续由权限拦截）
            filterChain.doFilter(request, response);
            return;
        }

        // 解析 token → 获取 userId → 从 Redis 取用户信息
        String userId = JwtUtil.parseJWT(token).getSubject();
        LoginUser loginUser = redisCache.getCacheObject("login:" + userId);

        // 写入 SecurityContextHolder
        UsernamePasswordAuthenticationToken authToken =
                new UsernamePasswordAuthenticationToken(loginUser, null, loginUser.getAuthorities());
        SecurityContextHolder.getContext().setAuthentication(authToken);

        filterChain.doFilter(request, response);
    }
}
```

### SecurityFilterChain 配置（JWT 版）

```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity   // 开启方法级权限注解
public class SecurityConfig {

    @Autowired
    private JwtAuthenticationFilter jwtAuthenticationFilter;

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .sessionManagement(session -> session
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS))  // 无状态
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/login", "/register").permitAll()        // 放行登录注册
                .anyRequest().authenticated()
            );

        // JWT 过滤器加在用户名密码过滤器之前
        http.addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }

    @Bean
    public AuthenticationManager authenticationManager(AuthenticationConfiguration config)
            throws Exception {
        return config.getAuthenticationManager();
    }
}
```

## 方法级权限控制

配置类加 `@EnableMethodSecurity` 后，在方法上直接声明权限：

```java
@PreAuthorize("hasAuthority('sys:user:list')")
@GetMapping("/user/list")
public Result list() { ... }

@PreAuthorize("hasRole('ADMIN')")
@DeleteMapping("/user/{id}")
public Result delete(@PathVariable Long id) { ... }
```

### 常用权限注解

| 注解 | 说明 |
|------|------|
| `@PreAuthorize` | 方法执行**前**校验，最常用 |
| `@PostAuthorize` | 方法执行**后**校验返回值 |
| `@Secured` | 基于角色校验（需单独开启） |

### `hasAuthority` vs `hasRole`

```
hasAuthority("sys:user:list")  → 精确匹配权限字符串
hasRole("ADMIN")               → 匹配 ROLE_ADMIN（框架自动加前缀）
```

## 常见问题

| 问题               | 原因 / 解决                                                                                            |
| ---------------- | -------------------------------------------------------------------------------------------------- |
| 403 Forbidden    | 已认证但无权限，检查 `getAuthorities()` 返回值是否正确                                                              |
| 401 Unauthorized | 未认证，检查 JWT 过滤器是否正确写入 SecurityContextHolder                                                         |
| 循环依赖             | `SecurityConfig` 注入 `UserDetailsService` 时，改用 `@Lazy` 或通过 `AuthenticationConfiguration` 获取 Manager |
| 密码匹配失败           | 确认数据库存的是 BCrypt 加密后的密码，而非明文                                                                        |

## 项目中具体的实现SecurityConfig
```Java
@Configuration  
public class SecurityConfig {  
  
    @Autowired  
    private JwtAuthenticationFilter jwtAuthenticationFilter;  
  
    @Bean  
    public PasswordEncoder passwordEncoder() {  
        return new BCryptPasswordEncoder();  
    }
  
    @Bean  
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {  
        http  
			.csrf(csrf -> csrf.disable())  
			.sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))  
			.authorizeHttpRequests(auth -> auth.anyRequest().permitAll())  
			.addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class);  
        return http.build();  
    }  
  
    // 关掉 Spring Boot 对 @Component Filter 的自动注册，避免被执行两次  
    @Bean  
    public FilterRegistrationBean<JwtAuthenticationFilter> jwtFilterRegistration(JwtAuthenticationFilter filter) {  
        FilterRegistrationBean<JwtAuthenticationFilter> registration
	         = new FilterRegistrationBean<>(filter);  
        registration.setEnabled(false);  
        return registration;  
    }  
}
```
### 1. `csrf(csrf -> csrf.disable())`

**作用**：关闭 Spring Security 的 CSRF 防护。

**原理**：

CSRF（跨站请求伪造）攻击针对的是「**基于 cookie/session 的认证**」——浏览器会自动带 cookie 到目标站点，攻击者诱导用户访问恶意页面就能伪造请求。

而 JWT 走 `Authorization` 头：
- 浏览器**不会自动**把 token 加到请求里，必须 JS 主动塞
- 攻击者无法伪造跨站请求

所以 CSRF 防护对 JWT 架构是多余的。

**不关会怎样**：Security 默认开启 CSRF 校验，所有 `POST / PUT / DELETE` 请求必须带 `_csrf` token，否则直接 403。

---

### 2. `sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))`

**作用**：禁止 Spring Security 创建 HttpSession，全程无状态。

**原理**：

Security 默认行为是登录成功后创建 `HttpSession`，存在服务端内存里，给浏览器返回 `JSESSIONID` cookie。下次请求带 cookie，从 session 里捞用户信息。

这种方式有两个问题：

| 问题 | 说明 |
| --- | --- |
| 多实例部署 | session 存在内存里，不同服务器之间不共享，需要粘性会话或 Redis 共享 session |
| 跨端兼容差 | 移动端、小程序没有 cookie 机制，session 玩不转 |

JWT 的设计目标就是**无状态**——用户信息全塞在 token 里，服务端不存 session。

> 不要给我创建 session，我用 JWT 自己管。

**为什么必须显式声明**：不写这行，Security 仍然会创建 session（用于登录失败计数、CSRF token 存储等）。这些 session 在 JWT 架构下是浪费 + 隐患。

---

### 3. `authorizeHttpRequests(auth -> auth.anyRequest().permitAll())`

**作用**：Security 链末端的「授权 Filter」对所有请求放行。

**原理**：

Security 链上有很多 Filter，`authorizeHttpRequests` 配置的是 Security **自己内置的** `AuthorizationFilter`（链的最末端）的规则。

而自定义的 `JwtAuthenticationFilter` 已经包揽了认证：

```
请求 → JwtAuthenticationFilter → ... → AuthorizationFilter → Controller
        ↑ 没 token / token 错              ↑ permitAll 放行
        直接 setStatus(401) 返回
```

- token 错 → 自定义 Filter 直接 401，**根本走不到** `AuthorizationFilter`
- token 对 → 自定义 Filter 放行，末端 `permitAll` 让它通过

**为什么不用 `authenticated()`**：

当前 Filter 验证完 token **没有**把用户信息塞进 `SecurityContextHolder`。Security 看不到"已认证"状态，会认为是匿名请求，`authenticated()` 直接拒绝。

> ⚠️ W2 做角色权限时，需要把用户塞进 `SecurityContextHolder`，再把这里改成 `authenticated()`，才能用 `@PreAuthorize` 做细粒度控制。

**面试答案**：

> 当前认证逻辑全部由自定义 JwtAuthenticationFilter 完成，验证失败直接返回 401。所以 Security 链末端的授权 Filter 设 `permitAll`——能走到这一步说明已经过了认证。后续做角色权限时，会把用户信息塞进 SecurityContextHolder，再用 `@PreAuthorize` 注解做细粒度控制。

---

### 4. `addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class)`

**作用**：把自定义 JWT Filter 插在 Security 内置的 `UsernamePasswordAuthenticationFilter` **前面**。

**原理**：

`UsernamePasswordAuthenticationFilter` 是 Security 内置的「表单登录 Filter」，专门处理传统的用户名密码 POST 表单。

把 JWT Filter 插在它前面，意味着 JWT 校验**先跑**：
- 没拿到合法 token → 直接 401
- 根本不会到表单 Filter

这是社区惯例位置，最不容易和其他内置 Filter 冲突。

---

### 5. `FilterRegistrationBean.setEnabled(false)`

**作用**：关闭 Spring Boot 对自定义 Filter 的 Servlet 容器自动注册。

**原理**：

自定义 Filter 加了 `@Component`，会触发**两套注册机制**：

| 注册路径 | 谁负责 | 范围 |
| --- | --- | --- |
| Servlet 容器自动注册 | Spring Boot | 对所有请求生效 |
| Security 链注册 | `addFilterBefore(...)` | 在 Security 链中生效 |

两套同时生效 → Filter 被注册两次 → 每个请求经过两遍。轻则浪费性能，重则解析两次 token、两次写 SecurityContext，引入诡异 bug。

**解决**：用 `FilterRegistrationBean` 包一层并 `setEnabled(false)`，告诉 Spring Boot **跳过** Servlet 容器自动注册，只保留 Security 链里的注册。

```java
@Bean
public FilterRegistrationBean<JwtAuthenticationFilter> jwtFilterRegistration(JwtAuthenticationFilter filter) {
    FilterRegistrationBean<JwtAuthenticationFilter> registration = new FilterRegistrationBean<>(filter);
    registration.setEnabled(false);
    return registration;
}
```

**面试答案**：

> 自定义 Filter 用 `@Component` 让 Spring 管理，但 Spring Boot 会自动把它注册到 Servlet 容器，加上 Security 链里又注册一次，导致 Filter 被执行两次。所以用 `FilterRegistrationBean.setEnabled(false)` 关掉 Servlet 容器的自动注册，让 Security 链全权负责。

## W1 阶段的局限 & 后续要补的

当前 SecurityConfig 处于 **能用但不完整** 的状态：

- ❌ 没把用户信息塞进 `SecurityContextHolder` → `@PreAuthorize`、`@AuthenticationPrincipal` 都拿不到当前用户
- ❌ 角色权限还没开启 → Owner / Admin / Member 区分还没实现

**W2 做角色权限时需要补的步骤**：

1. `JwtAuthenticationFilter` 解析完 token 后，查一次用户表，构造带角色的 `UsernamePasswordAuthenticationToken`，塞进 `SecurityContextHolder`
2. `permitAll()` 改成 `authenticated()` + 白名单
3. 配置类加 `@EnableMethodSecurity`，Controller 方法上用 `@PreAuthorize("hasRole('ADMIN')")`
