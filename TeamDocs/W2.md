# SecurityContext 
SecurityContext 本身只是一个接口，它的内部结构非常简单，核心就是为了持有 Authentication（认证对象）。
它们的关系就像套娃一样
```
SecurityContextHolder (大保箱/全局管理器)
└── SecurityContext (工牌夹/上下文)
└── Authentication (工牌/认证对象)
	├── Principal (用户主体，通常是 UserDetails)
	├── Credentials (凭证，如密码，通常在认证后被擦除)
	└── Authorities (权限/角色列表，如 ROLE_USER)
```
## 线程隔离：ThreadLocal

在默认情况下，SecurityContext 是通过 **ThreadLocal** 存储在 SecurityContextHolder 中的。

- **这意味着：** Spring Boot 采用的是“每个请求一个线程（Thread-per-request）”的模型。当一个请求进来时，框架会把当前用户的安全上下文绑定到当前线程。
    
- **带来的好处：** 在这个请求的生命周期内，无论你是在 Controller、Service 还是 Repository 层，都不需要把用户信息作为参数层层传递，直接从当前线程“隔空取物”即可。
    
- **请求结束：** 框架会在请求结束（Filter 链执行完毕）时，自动清理当前线程的 ThreadLocal，防止内存泄漏或线程复用导致的数据串台。
