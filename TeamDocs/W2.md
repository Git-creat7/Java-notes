# SecurityContext 
SecurityContext 本身只是一个接口，它的内部结构非常简单，核心就是为了持有 Authentication（认证对象）。
它们的关系就像套娃一样
```
SecurityContextHolder (大保箱/全局管理器)
└── SecurityContext (工牌夹/上下文)
└── Authentication (工牌/认证对象)
	├── Principal (用户主体，通常是 UserDetails)
	├── Credentials (凭证，如密码，通常在认证后被擦除)
	└── Authorities (权限/角色列表，如 ROLE_USER)
```
## 线程隔离：ThreadLocal

在默认情况下，SecurityContext 是通过 **ThreadLocal** 存储在 SecurityContextHolder 中的。

- **这意味着：** Spring Boot 采用的是“每个请求一个线程（Thread-per-request）”的模型。当一个请求进来时，框架会把当前用户的安全上下文绑定到当前线程。
    
- **带来的好处：** 在这个请求的生命周期内，无论你是在 Controller、Service 还是 Repository 层，都不需要把用户信息作为参数层层传递，直接从当前线程“隔空取物”即可。
    
- **请求结束：** 框架会在请求结束（Filter 链执行完毕）时，自动清理当前线程的 ThreadLocal，防止内存泄漏或线程复用导致的数据串台。

## UsernamePasswordAuthenticationToken
它继承自 AbstractAuthenticationToken，核心存储了三个核心要素：

- **principal (主体)：**
    
    - 在 _认证前_，它通常是一个简单的 String（用户输入的用户名）。
        
    - 在 _认证后_，它会变成你的 UserDetails 框架实体类（比如你自定义的 LoginUser），方便你在业务层直接获取用户 ID、手机号等扩展属性。
        
- **credentials (凭证)：**
    
    - 通常就是密码。处于安全考虑，Spring Security 在认证成功将 Token 放入 SecurityContext 之前，默认会自动调用 eraseCredentials() 方法**把密码擦除（设为 null）**，防止密码在内存中常驻。
		
	> 类比：你刷门禁卡进了公司，进门那一刻卡的"作用"已经完成了，公司大堂的访客系统不需要再记着"这人卡号是 12345"。它只需要记着"工号 1001 的张三现在在公司里"——这个就是principal（你的 LoginUser）
        
- **authorities (权限集合)：**
    
    - 当前用户拥有的角色（如 ROLE_ADMIN）或具体权限权限（如 sys:user:delete），后续框架的 @PreAuthorize 注解就是基于它来鉴权的。
```
 [用户请求]──>(UsernamePasswordAuthenticationFilter)
			 
			│
			▼ 1. 提取账号密码，创建【未认证 Token】
			(AuthenticationManager)
			
			│
			▼ 2. 校验密码，去数据库查 UserDetails
			
			(AuthenticationProvider)
			
			│
			▼ 3. 校验成功，创建【已认证 Token】（带权限）

[放入上下文]<──(SecurityContextHolder)
```
